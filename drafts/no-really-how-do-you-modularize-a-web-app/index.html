No, Really, How do you modularize a web app?

Written by Jordan H.

The goal of every web application is to be as modular as possible. But to the novice developer, it can be hard to wrap your head around what this actually means. There are simple examples on YouTube of how this works (your bound to see simple to-do list apps), but what does this mean at scale? In this article I describe in detail what it means to modularize your web app.

Don’t Think Like an Assembly Line

Let’s first use an illustration. Imagine our server is a set of these happy androgenous blue assembly line workers. Their job is pretty simple. The first person makes the box, the second person adds a doohicky on the box, the third person puts a thingamabob on the doohicky, and a the last person puts some sort of bat wings on top.

What is this thing? Who knows? But for our example it’s our web response.

Now, everything is working perfectly and the response is getting shipped out each time.

But one worker along the assembly line doesn’t feel too good. What’s got him down? Who knows! But in any case he has to go home.

Now what happens? Let’s see.

As you can see it’s complete chaos. Nothing is working right. The factory explodes. People are out of work. That’s not good.

This is how to poorly design a web server. You see, everything is very tightly interconnected, and one thing relies on the next. If one part of the web app is broken, the entire system from then on doesn’t work!

Instead, think Like a River

2 By 2 Hands of Blue?

No, not that River! A waterway.

Imagine a fisherman (let’s name him Bob) at the bottom of the stream hoping to catch a web response–er, salmon.

His evil twin Jerry doesn’t want him to catch fish. Because Jerry’s evil. So to screw up his brother’s dinner plans he steals a boulder from the river (he’s also very strong) and carries it away.

But Bob doesn’t notice anything different? Why’s that?

Because a river is more resilient than an assembly line. It finds a new path to Bob so that he can have that tasty salmon.

Likewise, we can design our web server like this.

Web Framework

This is why a web framework is helpful. They’re designed to be resilient (at least the good ones) are designed to be this way.

Each framework is different. I’ll just close my eyes hold out my finger and point to any ol’ framework. Any. one… Oh, just happens to be Django, one of my favorite web frameworks.

If you’ve never used Django, head on over to https://djangoproject.org and read up on the project. If you’ve used other web frameworks before it’s pretty similar.

Apps

The framwork is divided into apps. Each app can contain the following components:

  • Python module, which contain
    • views, or
    • models, or
    • both, or
    • additional magic
  • templates
  • template tags and filters
  • URLs

Note, there may be more components, such as utility methods meant to be shared with other apps. And also note apps don’t necessarily contain all of these.

Apps are designed to be reusable, modular, and extensible.

So you’ve seen what’s already in the industry. Let’s get our hands dirty and create a project with an app that’s modular.

I’ll skip a lot of the nitty-gritty details since I want the focus to be very high-level and framwork-independent.

Nothing to fancy here. We’re just starting with a blank Django project.

Let’s add 2 Features

:A Menu: This will be a simple model that will have link text and a url and can be implemented as a template filter.

:A Sidebar: Just contains simple sidebar HTML to be displayed. This will also be implemented as a template filter.

These will not be separate apps as listed above, but baked into the application, following a monolithic approach.

Menu

We’ll basically have a “Menu” model, with a “Link” model that contains a 1 → N relationship between Menu and Link.

Then to display the menu in the project we’ll just call the menu by name.

Sidebar

We’ll do something similar to Menu here. However, we just have one sidebar and no links associated with it.

The Result

So far so good. So what’s the issue?

Well, there are several issues with this.

1. We Can’t Reuse the Code

At least not very easily. Unless you’re a HUGE fan of copy and paste. Even if you were, if you change the code in one project you’d have to make sure the code was updated across all your projects.

2. We Can’t Remove Components Easily

Since the compnents are baked into our application, if we want to remove it (let’s say we need to do some maintenance).

3. Updating one component updates the entire application.

This can raise a whole host of problems, such as miscommunication between developers and merge conflicts.

Let’s Try Refactoring

Questions To Ask

So, obviously I’ve already spilled the beans on this one by saying I’ll have a menu and a sidebar component. So I don’t have to do a whole lot of brainstorming to figure out how to refactor this.

But let’s say I just plunged into building this application without planning to make it modular. There are some questions I can ask myself to decide if a component or feature is worthy of being bestowed with the powers of modularity:

  • Is this separate from the overall goal of the application?
  • Can this component be reused in another part of the application?
  • Do I find myself copying and pasting code segments a lot?
  • If I took this class/method/whatever out of the code, can I swap in something else?

Hopefully you get the idea. Just think of your program like a car until the early 2000’s. They were meant to be repaired. Parts were begging to be swapped out to get you on the road fast.

Refactoring

When making these components modular, it helps to think about the program flow. So if you’re just starting with a new framework I’d recommend doing it the “wrong” way for a bit to get a handle on how things work.
